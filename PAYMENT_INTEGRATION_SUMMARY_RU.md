## 📊 Что было построено

### 1. **Интеграция Stripe Checkout** ✅
- Сессии создаются на бэкенде с ключами идемпотентности
- Фронтенд перенаправляет на Stripe checkout
- Настроены URLs для успеха и отмены
- Режим тестирования готов (переключите на production ключи при развёртывании)

### 2. **Обработка Webhooks** ✅
- Проверка подписи на каждом событии
- Идемпотентная обработка (повторы безопасны)
- Два типа событий: `checkout.session.completed` → оплачено, `checkout.session.expired` → не удалось
- Отложенное использование промо: создаётся только после webhook подтверждения оплаты

### 3. **Предотвращение дублирования заказов** ✅
- **Унифицировано для ВСЕХ методов оплаты** (Stripe, наличные, карта)
- SHA256 отпечаток нормализованного payload
- Окно повторного использования: 30 минут
- Для Stripe: переиспользует сессию если открыта; отмечает как failed если истекла
- Для наличных/карты: возвращает существующий заказ вместо создания дубликата

### 4. **UX при отмене Stripe** ✅
- Черновик формы хранится в sessionStorage
- При отмене возвращает на checkout с автозаполненной формой
- Баннер с объяснением что произошло
- Корзина НЕ очищается до успешного возврата
- Успешный возврат на `/track/[token]?paid=1` запускает очистку

### 5. **UI статуса оплаты** ✅
- Отдельная строчка `Статус оплаты` на странице отслеживания
- Три состояния: ⏳ Ожидание (pending), ✅ Оплачено (paid), ❌ Не оплачено (failed)
- Цветные значки (жёлтый/зелёный/красный)

### 6. **Защитные механизмы** ✅
- Отклонение заказов с нулевой суммой для Stripe (предотвращает broken сессии)
- Отложенное использование промо для Stripe (защита от пропадания при отмене)
- Проверка минимальной суммы для зоны доставки
- Проверка минимальной суммы для промо-кода
- Ограничения БД: stripe_session_id уникален, fingerprint индексирован

## 📝 Ключевые детали реализации

### Состав отпечатка
```typescript
{
  type,
  payment_method,
  customer_name,
  customer_phone,
  address fields...,
  promo_code,
  language_code,
  total_amount,
  items (нормализованные и отсортированные)
}
```
→ SHA256 хеш → используется для обнаружения дубликатов

### Поток событий Webhook
```
Stripe → POST /api/stripe/webhook (подпись проверена)
  ↓
checkout.session.completed
  ├─ Найти заказ по metadata.orderId
  ├─ Проверить session.payment_status === "paid"
  ├─ Отметить order.payment_status = "paid"
  ├─ Создать PromoCodeUsage (если есть promo_code_id)
  └─ Вернуть 200 OK
  
checkout.session.expired
  ├─ Найти заказ по metadata.orderId
  ├─ Отметить order.payment_status = "failed"
  └─ Вернуть 200 OK
```

### Логика обнаружения дубликатов
```
1. Вычислить checkout_fingerprint из запроса
2. Запрос: есть ли pending заказ с одинаковым отпечатком + метод оплаты за последние 30 мин?
   ДА (Stripe):
     - Получить сессию из Stripe API
     - Если открыта: вернуть URL существующей сессии
     - Если истекла: отметить как failed, продолжить создание нового заказа
     - Если complete & paid: вернуть ответ об успехе
   ДА (Non-Stripe):
     - Вернуть существующий заказ (новый платёж не нужен)
   НЕТ:
     - Создать новый заказ
     - Для Stripe: создать сессию с idempotencyKey
     - Для остальных: готов сразу
```

## 🐛 Распространённые проблемы и решения

### Проблема: статус оплаты показывает "pending" после платежа Stripe
**Решение:** Webhook может задержаться на 1-3 секунды. Обновите страницу или подождите.

### Проблема: заказ создан но платежа нет в Stripe
**Решение:** Проверьте логи webhook. Если отсутствует checkout.session.completed, переотправьте из Stripe dashboard.

### Проблема: миграция БД не применяется
**Решение:** Проверьте что PostgreSQL запущен. Верифицируйте CONNECTION_STRING в DATABASE_URL.

### Проблема: тестовая карта Stripe отклоняется
**Решение:** Используйте только `4242 4242 4242 4242` для тестов. Никогда не используйте реальные карты!

## 🎓 Архитектурные решения

**Почему Stripe только на бэкенде?**
- Secret ключи никогда не видны фронтенду
- Проверка подписи webhook всегда надёжна
- Логика переиспользования сессии под нашим контролем

**Почему отпечатки для ВСЕХ методов?**
- Защита от случайных API retries для ВСЕХ workflow
- Единообразный UX: пользователь никогда не видит "заказ уже создан"
- Будущепруф: любой новый метод платежа автоматически защищен

**Почему деferred промо для Stripe?**
- Отменённые Stripe не тратят квоту промо
- Failed webhook не используют промо
- Transaction-safe: промо отмечено только после подтверждённого платежа

**Почему sessionStorage для черновика?**
- Пережидает перезагрузку страницы + switch tab
- Очищается при успехе (через TrackClientState useEffect)
- Не сохраняется при закрытии браузера (хорошо для приватности)

## 🚨 Известные ограничения

1. **Только режим теста Stripe:** На данный момент настроен для test API. Production требует замены ключей.
2. **Наличная оплата не автоматизирована:** Админ должен вручную отметить как оплачено в меню заказов (будущее: страница подтверждения вручную).
3. **Нет UI возвратов:** Возвраты Stripe обрабатываются через Stripe dashboard (будущее: интеграция в админ-панель).
4. **Одна валюта:** Все цены в EUR. Поддержка множественных валют требует пересчёта промо.
5. **Повторная отправка webhook вручную:** Зависит от Stripe dashboard (будущее: экспоненциальная задержка в бэкенде).
